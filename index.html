        Wisely API

        Wisely provides an API to interact with the core functionality of our platform and to support your brand's existing guest data integrations.

        Dates / Timestamps

        • All datetimes should be submitted in ISO8601 format and in UTC, unless otherwise stated for a specific property. See request bodies for a specific request for examples.

        Error Codes

        401: Unauthorized

        • Ensure the request contains a valid, non-exired bearer token

        404: Not found

        • Couldn't locate the resource you're seeking

        422: Validation error

        • Check the payload to ensure the values are valid Example:
        {
  "message": "A validation error happened",
  "validation_errors": {
    "location.id": "Invalid value",
    "order.opened_at": "Invalid value",
  }
}

        429: Rate limited

        • Use the response headers to decide when to submit the next request

        500: Internal server error

        • The server experienced an unexpected error. Please contact Wisely for further assistance

        Reservations

        Wisely provides a drop-in HTML snippet to display the Reservations flow on your website. If you would like to create a more custom experience, our API allows you to perform all of the same actions.

        Using this API you will be able to query reservation availability and create, update and delete reservations. Along with reservations you can attach any custom fields that are unique to your custom experience.

        Authorization

        This API will require a OAuth Bearer Token. Omitting the token will result in a 401 Unauthorized Response. See Get Bearer Token for details.

        GET

        Reservation Availability

        {{baseUrl}}/reservations/availability?merchant_id=12345&party_size=5&search_ts=2020-05-11T18:15:00-04:00&search_window_minutes=180

        Reservation availability is broken down into availability_types. You will have one default type and for more interesting use cases, you may add additional options by contacting your Wisely CSM. By default, the API will return an array containing a single availability_type denoted with is_default=true. Please see the example response provided below.

        Within the availability type object, you will see a slots array containing objects (timestamp, is_available). Each object represents a time slot that will accept additional bookings for the provided party_size

        search_ts represents the guest's desired reservation time. For this reason we sort the available time slots in descending order by proximity to this timestamp. We recommend displaying multiple time slots to your guests when they search to increase the likelihood they will find an available time slot on a busy night, thereby increasing conversion.

        Depending on how your user interface behaves, you may want to search for availability over the entire day or constrained to a smaller window. By default, the API will return available slots for the entire day. To constrain the search, use the parameter search_window_minutes. For example, passing search_window_minutes=180 would constrain the search results to 3 hours on either side of your provided search_ts.

        Orders

        Authorization

        This API will require a OAuth Bearer Token. Omitting the token will result in a 401 Unauthorized Response. See Get Bearer Token for details.

        PUT

        Create Order

        {{baseUrl}}/orders

        Description

        Use this request to save order, ticket, and guest data to Wisely.

        Conceptually, an order contains an array of tickets. Each ticket must have a customer object, which represents who ordered the items on the check

        Rate Limit: Soft limit of 30 requests per second per store.


        Request body

        order (required | object)

        key type description
        order.source_id (required) string The source ID of the order, per the POS / ordering provider. Must be completely unique across all systems
        order.location_source_id (required) string The source ID of the location that's fulfilling the order. Must exist in Wisely prior to saving data
        order.source_type (required) string The provider of the order data. Currently, only accepted values are "toast" or "olo"
        order.delivery_method string The manner in which the order was delivered. Ex. curbside, delivery, dispatch, pickup
        order.delivery_address object Information about where the order is being delivered. Fields include service_address_1, service_address_2, city, zip_code, comments, a coordinates object. coordinates contains a latitude property and a longitude property, both in decimal formats.
        order.opened_at (required) datetime Date time of when the order was opened. Submitted in UTC
        order.ready_at datetime Date time of when the order is ready to be picked up by a guest. Submitted in UTC
        order.cancelled_at datetime If the order was cancelled, this field represents when the order was cancelled. Submitted in UTC
        order.closed_at datetime Date time of when the order was closed. Submitted in UTC
        order.pickedup_at datetime Date time of when the order was picked up by the guest. Submitted in UTC
        order.name string The name of the order, likely entered via the POS or the ordering provider
        order.provider_id string The provider by which the order was made through. Examples include toast, grubhub, doordash, etc.
        order.tickets (required) array Represents a single ticket / check in the POS. Includes data like the items, and payments on the ticket. Also includes data bout the customer who paid out the ticket
        ticket.source_id (required) string The source ID of the check, per the POS / ordering provider. Must be unique per location_source_id
        ticket.open (required) boolean Represents whether the check is open and hasn't been paid out yet, or if it's closed
        ticket.guest_count number Represents how many guests were on the check, if any
        ticket.closed_at datetime When the ticket was closed and paid out in the POS. Submitted in UTC
        ticket.name string The name on the ticket, as entered by the server or integration via the POS
        ticket.table_id string The ID of the table that the guest has been sat at, if applicable
        ticket.revenue_center_id string The ID of the revenue center that the ticket was created in. Will be specific to the POS, if the POS has the concept of a revenue center
        ticket.employee_id text The ID from the source_identifier which identifies the employee who serviced the ticket
        ticket.voided (required) boolean Whether or not the ticket was voided and not counted as sales in the POS
        ticket.ticket_number number The number displayed on the POS for the specific ticket
        ticket.paid (required) number The total amount paid by the guest, in case of partial payments. Value is in cents
        ticket.tip (required) number The total amount tipped by the guest. Value is in cents.
        ticket.tax (required) number The tax amount due on the check. Can vary by state. Value is in cents.
        ticket.due (required) number The amount due for the guest to pay when checking out. Value is in cents.
        ticket.sub_total (required) number The sub total on the check, considering item sales and discounts. Value is in cents
        ticket.total (required) number The total on the check, considering the sub total and tax. Value is in cents
        ticket.customer (required) object The customer who is purchasing the items on the ticket
        customer.email (required) string The email of the customer. This is required and is used to uniquely identify a user
        customer.phone string The phone number of the customer, expected in the e164 format
        customer.fname (required) string The first name of the customer
        customer.lname string The last name of the customer
        customer.email_marketing_opt_in boolean To opt in the customer to Wisely's email marketing.
        customer.sms_marketing_opt_in boolean To opt in the customer to Wisely's SMS marketing.
        ticket.payments (required) array You must speak with our customer support to ensure that you are eligible to use this. All payments made on the current ticket. Can be empty if no payments have been made yet
        payment.source_id (required) string The ID of the payment, based on the integration
        payment.amount (required) number The amount paid by the guest. Value is in cents
        payment.tip (required) number The amount tipped by the guest. Value is in cents
        payment.last4 string The last 4 of the credit card used on that payment. If the payment was made via cash, this field should be null
        payment.payment_type (required) string The type of payment, must be one of the following values: [credit_card, debit_card, ebt_card, gift_card, google_pay, apple_pay, cash]
        payment.card_type string What type of payment the card is (VISA, MasterCard, etc.)
        payment.comments string Any free field notes made on the payment
        payment.card_token string If the card token integration is enabled for the provider, this field represents the unique card token for the credit card on this payment. If the payment was cash, this field should be null
        ticket.items (required) array All line items on the check that the guest will have to pay for
        item.source_id (required) string The ID of the item, based on the integration
        item.name (required) string The name of the item, as represented by the restaurant's menu
        item.comment string A free text field present if the guest adds special requests onto the item
        item.price (required) number The total price of the item at the given quantity after discounts have been applied, if any. Value should be in cents
        item.quantity (required) number The quantity of items being purchased
        item.voided (required) boolean Represents whether the item was voided or not
        item.menu_item_id (required) string The ID of the menu item, per the restaurant's menu
        item.modifiers (required) array All modifiers which modify the current item in question. Ex. "add onions" on a hamburger. Can be an empty array
        modifier.source_id (required) string The ID of the modifier, based on the integration

        Guest Data Streams

        Each data source will have a unique Source Token that identifies you as a customer as well as the specific source of data. For example, a loyalty provider, POS data, and raw HTTP requests will each have their own token.

        This allows us to forward data to specific destinations and intelligently control how identities are merged. Keeping the tokens separate also allows forwarding historical data from a specific source if you choose to switch destination vendors in the future (eg. switching to a different marketing provider)

        Authorization

        The Data Streams API uses basic authentication. Omitting the token will result in a 401 Unauthorized Response.

        HTTP Basic Authentication involves username:password, base64 encoded, and prepended with Basic.

        In practice for this API, you will take a Source Token, append a colon, and leave the password empty. For example, if your token is abcde12345, you will base64 encode it to become YWJjZGUxMjM0NQ==. You will then pass this as the authorization header like so: 'Authorization: Basic YWJjMTIzOg=='.

        Webhooks and Third Party Destinations

        All event and identity data captured by the Data Streams API can be forwarded to a destination of your choosing via a POST request to a configurable URL. This data will be sent formatted exactly how the original request was made. See Webhooks for more details.

        Wisely also supports direct integrations to various data destinations like loyalty and marketing providers. This will empower your brand to utilize combined guest data in any platform of your choosing without vendor lock in. Please discuss the integration options with your Wisely CSM.

        Webhooks

        Webhooks allow Wisely to notify a custom application when events happen. This might include when order data is collected, a guest arrives to pickup their order, or a guest leaves feedback. Wisely simply forwards the data to your endpoint so you can perform any custom logic you'd wish!

        The configured Webhook Endpoint must return a 200 response as quickly as possible. Please return a success before performing any complex, time consuming logic on your system.

        Retries

        Wisely will attempt to send each event up to 3 times if we receive a 5xx response. We employ a basic exponential backoff algoritm for retries (eg. 1 second, 4 seconds, 9 seconds)

        Get Bearer Token

        {{baseAuthUrl}}/oauth2/default/v1/token

        Using your Wisely provided client ID and client secret, you may fetch a refreshed access token. Your credentials carry access to your customer data, so be sure to keep them secret! Do not share your client_id or client_secret in publicly accessible areas such as GitHub or client-side code.

        The access token is a JWT containing the keys token_type, expires_in, access_token, and scope.

        You will use the access_token key prefixed with "Bearer " in the Authorization header for all requsts. Authorization: Bearer eyJraWQiOiIx...

        **Please note expires_in indicates that the token is valid for 3600 seconds (1 hour). Please cache your tokens for this duration before refreshing or you may experience a rate limit.
